01 //

任務簡報

教科書告訴你 async 比較快、pool 要設大一點。這裡讓你量出來——而且常常量出和直覺相反的結果。

為什麼需要一台靶機？

壓測最常見的失敗不是「不會用工具」，而是分不清瓶頸在誰身上： 你以為在測伺服器，其實在測攻擊機的 CPU；你以為 async 是魔法， 其實 pool 太小時 sync/async 根本沒差。

靶機把每個變因做成可旋轉的旋鈕：同一個端點有 sync / async 兩個版本（同 driver、同 pool，唯一變因是併發模型）、 DB_POOL_SIZE 控制連線池、 ?delay_ms= 模擬慢查詢。 一次只轉一顆，差異才能歸因。

攻防隔離：三容器 × cpuset

攻擊機（Locust）打到火力全開時 CPU 會吃滿 100%。如果它和靶機擠在同幾顆核心上， 你量到的延遲一半是攻擊機自己造成的。所以整套架構容器化，用 cpuset 把三方鎖在不同核心，互不搶資源。

db：防守方的資料庫 app：靶機——給 2 核是刻意的（sync threadpool 可跨核，async event loop 只吃得到 1 核，不對稱本身就是考點） locust：攻擊機——給最多核，它不能先成為瓶頸

六連 Demo 課表

順序是設計過的：先學工具（1）→ 先懂 event loop（2）→ 先排除攻擊機瓶頸（3）→ 才做靶機實驗（4、5）→ 最後整合大流量（6）。 跳過 3 直接做 4，你測到的會是 Locust 的極限，不是靶機的。每支腳本的 docstring 都有完整 runbook。

01 locustfiles/demo1_basics.py

Locust 入門 — User / @task / wait_time / Web UI

 第一份 p50/p95 報表；RPS ≠ users 數▶

三個核心概念：User（一個虛擬使用者 = 一個 greenlet）、@task(weight)（任務與權重）、wait_time（思考時間，真人不會連發請求）。預設腳本就是它，全套起完直接開打：

# 全套起來（db + 靶機 + Locust）
docker compose up -d --build --wait
open http://localhost:8089   # 10 → 50 → 100 users 各跑一輪

要能回答：100 users + wait_time(1,3) 的理論 RPS 上限是多少？（100 ÷ 平均 2s ≈ 50 —— RPS 不是 users 數！）

Event Loop 卡死實況 — 實驗 A / B / C

03 locustfiles/demo3_attacker_bottleneck.py

攻擊機自己先陣亡 — HttpUser vs FastHttpUser

300 users 打 /healthz（靶機最快的端點）。同參數下 HttpUser 5,921 RPS、FastHttpUser 24,762 RPS——差的不是靶機，是攻擊機自己的 CPU。Web UI 開了 --class-picker，勾選一個 class 跑一輪，對照。

04 locustfiles/demo4_sync_vs_async_db.py

sync vs async 打真 PG — 核心對照組

 同 100 併發：async 512 RPS vs sync 335 RPS▶

同 driver（psycopg3）、同 pool（QueuePool），唯一變因是 create_engine vs create_async_engine。sync 端點佔住 threadpool 的 thread（預設 40 條）等 DB；async 端點等 DB 時 event loop 切去服務別人。

# 開大 pool（排除 pool 瓶頸）。一定要 export —— 只當前綴的話，
# 下一個 compose 指令會用預設 pool=5 把 app 默默重建回去（真實踩過）
export DB_POOL_SIZE=100
docker compose up -d app
LOCUSTFILE=demo4_sync_vs_async_db.py docker compose up -d locust

要能回答：sync 版的瓶頸是 threadpool 40 條還是 pool 連線數？怎麼設計實驗分開驗證？

05 locustfiles/demo5_pool_starvation.py

Pool Starvation — DB_POOL_SIZE 10 → 100 掃描

 167 → 642 RPS：pool 翻倍 p95 砍半，直到 CPU 102%▶

用 ?delay_ms= 讓查詢佔住連線，掃描 pool 大小看 p95 怎麼走。實測曲線（200 users / delay_ms=50）：pool=10/20/50/100 → RPS 167/302/620/642、p95 1400/740/390/420ms——pool 翻倍 p95 砍半，直到靶機 CPU 102%、瓶頸轉移到 event loop 為止。pool=10 時 sync/async 幾乎沒差（1.28s vs 1.28s）——瓶頸在誰身上，誰就決定 p95。每輪之間重置 DB，seed 一致數據才可比：

docker compose down
export DB_POOL_SIZE=10    # 下一輪換 20 / 50 / 100；要 export，理由同 demo4
docker compose up -d --wait
LOCUSTFILE=demo5_pool_starvation.py docker compose up -d locust

要能回答：pool_timeout 30s 到了會發生什麼？demo6 看到的某些 error 就是它。

06 locustfiles/demo6_step_load_social.py

階梯大流量 — 10 → 100 → 1000 → 5000 混合流量

 1000 users 撐住、5000 崩：p95 = 30s pool timeout▶

整合戰：讀寫混合流量階梯式上量，找出靶機的崩潰點。實測（pool=100）：1000 users 撐住（CPU 56-69%、零 error）；5000 users 崩潰——CPU 100%、MEM 貼死 256MiB limit、HTTP 500 ×2,802（3.23%），p95 = 30 秒整 = QueuePool limit reached, timeout 30.00。這條曲線就是階段三 HPA 要對抗的敵人。容器的 fd 上限（nofile=10240）已在 compose 設好。

LOCUSTFILE=demo6_step_load_social.py docker compose up -d locust
docker stats   # 三個容器一起盯：誰先死？

要能回答：崩潰時的第一個 error 是什麼類型（pool timeout？connection refused？OOM？）——它指向哪一層？

Runbook — 如何運行

# 一鍵全上：db + 靶機(:8000) + Locust(:8089)，cpuset / 記憶體限制全套生效
docker compose up -d --build --wait

# 換 demo 腳本（蓋掉預設的 demo1）
LOCUSTFILE=demo4_sync_vs_async_db.py docker compose up -d locust

# 實驗旋鈕：pool 大小
DB_POOL_SIZE=100 docker compose up -d app

# 每輪重置 DB（不掛 volume，down 即回到同一份 seed：50 users / 200 posts）
docker compose down && docker compose up -d --wait

# headless 一次性跑法（沿用同一組 cpuset / 記憶體限制）
docker compose run --rm locust \
  -f /mnt/locust/demo1_basics.py --host http://app:8000 \
  --headless -u 50 -r 10 -t 60s

# 鐵則 1：同時盯攻防兩側的 CPU / MEM
docker stats
# 攻防同擠一台機器會互搶 CPU（demo3 的教訓）—— 開發、改 code 時用
docker compose up -d --wait db        # 只起 PostgreSQL（host port 5433）
uv sync
uv run main.py                        # hot-reload，http://127.0.0.1:8000

# 本機壓測：關 reload、固定 1 worker、關 access log
ulimit -n 10240
uv run uvicorn app.main:app --host 0.0.0.0 --port 8000 --no-access-log
uv run locust -f locustfiles/demo1_basics.py
鐵則 01同時監控攻擊方與防守方（CPU / MEM）——分不清瓶頸在誰身上的數據沒有意義。容器模式一行搞定：docker stats
鐵則 02一次只改一個變因：users 數、pool 大小、sync/async——不要同時動，動了就無法歸因。
鐵則 03每輪重置 DB：docker compose down && up，每輪都從同一份 seed 開始，數據才可比。
鐵則 04用 Problem-Driven 格式記錄每一輪實驗——寫不出 Lesson 的實驗等於沒做：
Problem:    （觀察到什麼異常／想驗證什麼）
Hypothesis: （你認為的原因）
Experiment: （改了什麼、攻擊參數）
Result:     （p50 / p95 / RPS / error rate / 兩側 CPU·MEM）
Lesson:     （一句話，能教給別人的那種）

兩支探針，兩個問題

livenessProbe → /healthz：「死了嗎？」失敗就重啟 container。 刻意不碰 DB——如果 liveness 打 DB，pool 耗盡時 K8s 會誤判 App 死了而瘋狂重啟， 那不是 App 的問題，是探針設計錯誤。

readinessProbe → /readyz：「準備好了嗎？」真打一次 DB， 失敗只是暫時從 Service 摘掉、不重啟。DB 還沒好，Pod 就不收流量。

demo2 的 K8s 進階版：sync-sleep 卡死 event loop → 連 /healthz 都回不了 → liveness 連續 fail → 重啟 → 新 Pod 起來又被打 → 又重啟——這就是 K8s 下的雪崩迴圈。 模擬器（03）演的那一幕，在 K8s 裡會被 kubelet 放大。

單 Pod baseline：押注 → 跑 → 對答案

demo4 async・100u・delay 100ms・pool 100RPSp50
compose（cpuset 隔離）512200ms
K8s 單 Pod（CFS quota）440180ms

RPS 低 ~14%、p50 反而略好——差異來自攻擊機在 host 與 VM 共擠 + kube-proxy/LoadBalancer 多一跳 + CFS 無核心隔離。靶機 CPU 648m，離 limit 2000m 還遠。 量級一致 → 單 Pod baseline 成立，「1000 撐住、5000 必崩」就是階段三 HPA 的對照組。

踩坑 01 · 真實案發K8s 的 command: 蓋掉的是 ENTRYPOINT，args: 才對應 Docker 的 CMD。把 compose 的 command 原樣搬過來 → docker-entrypoint.sh 被跳過 → init.sql 沒跑、PG 以 root 啟動被拒 → db 和 app 一起 CrashLoopBackOff。
踩坑 02 · 真實案發RollingUpdate 讓 DB 連線足跡翻倍。新舊 Pod 並存的瞬間，pool 預熱撞上 too many clients → 新 Pod CrashLoop、舊 Pod 不退 → rollout 永久死鎖。本 demo 用 Recreate 治標，production 正解是 pgbouncer（→ LESSON 05）。
踩坑 03改 ConfigMap 不會自動生效。configMapKeyRef 的 env 只在容器啟動時讀一次，必須 kubectl rollout restart 才收斂——驗證方式：kubectl exec ... env | grep DB_POOL_SIZE（LESSON 04 的 K8s 版：先驗證旋鈕接上了）。
踩坑 04kubectl patch 是暫時的。patch 是 imperative 操作，下次 kubectl apply 會把 ConfigMap 蓋回 YAML 宣告的值——declarative 才是真相，實驗完想保留設定就改 manifest 再 apply。
# image 先 build（OrbStack K8s 直接看得到，免推 registry）
docker compose build app

# apply 有依賴順序：namespace 先行
kubectl apply -f k8s/namespace.yaml
kubectl apply -f k8s/

# 等 Pod Ready，拿 LoadBalancer 的 EXTERNAL-IP
kubectl get pods -n loadtest -w
APP_IP=$(kubectl get svc app -n loadtest -o jsonpath='{.status.loadBalancer.ingress[0].ip}')
curl http://$APP_IP:8000/healthz

# host 上開打（換 demo 只換 -f，不用重新部署靶機）
uv run locust -f locustfiles/demo4_sync_vs_async_db.py --host http://$APP_IP:8000 \
  --headless -u 100 -r 10 -t 60s

# 鐵則 01 的 K8s 版：盯資源用量
kubectl top pod -n loadtest

# 實驗旋鈕：pool 大小（改完必須 rollout restart 才生效——踩坑 03）
kubectl patch configmap app-config -n loadtest --patch '{"data":{"DB_POOL_SIZE":"100"}}'
kubectl rollout restart deployment/app -n loadtest
kubectl exec -n loadtest deploy/app -- env | grep DB_POOL_SIZE   # 驗證旋鈕接上了

LESSON 01瓶頸在誰身上，誰就決定 p95。pool 小的時候 sync/async 沒差——先找到瓶頸，再談優化。
LESSON 02攻擊機也會是瓶頸。xargs+curl 起 100 個 process 的 fork 開銷就吃掉 0.3s，差異完全被淹沒——所以 Locust 要用 FastHttpUser，所以容器要分核。
LESSON 03ORM 不是免費的。200 筆 row → ORM 物件的序列化是 CPU 工作，async 版全擠在單一 event loop 上無法重疊——抽象層的成本，async 要自己扛。
LESSON 04改了變因卻沒變化？先驗證旋鈕真的接上了。沒 export 的 DB_POOL_SIZE 被下一個 compose 指令默默重設回 5，掃了四輪全是同一份假數據——差點寫成教材。每輪跑完 docker inspect 驗 env。
LESSON 05零停機部署會讓 DB 連線足跡翻倍。K8s RollingUpdate 新舊 Pod 並存的瞬間，pool 預熱撞上 too many clients → rollout 永久死鎖——壓測靶機改用 Recreate，production 用 pgbouncer。
