01 //

任務簡報

教科書告訴你 async 比較快、pool 要設大一點。這裡讓你量出來——而且常常量出和直覺相反的結果。

為什麼需要一台靶機？

壓測最常見的失敗不是「不會用工具」，而是分不清瓶頸在誰身上： 你以為在測伺服器，其實在測攻擊機的 CPU；你以為 async 是魔法， 其實 pool 太小時 sync/async 根本沒差。

靶機把每個變因做成可旋轉的旋鈕：同一個端點有 sync / async 兩個版本（同 driver、同 pool，唯一變因是併發模型）、 DB_POOL_SIZE 控制連線池、 ?delay_ms= 模擬慢查詢。 一次只轉一顆，差異才能歸因。

01 locustfiles/demo1_basics.py

Locust 入門 — User / @task / wait_time / Web UI

 第一份 p50/p95 報表；RPS ≠ users 數▶

三個核心概念：User（一個虛擬使用者 = 一個 greenlet，跑自己的任務迴圈）、@task(weight)（任務與權重，read_posts(3):read_users(1) = 75%:25%）、wait_time（思考時間，真人不會連發請求）。

實驗設計 — 這支還沒有對照組，先學會「讀報表」
固定不動端點 /async/posts:/async/users = 3:1、wait_time between(1,3)、靶機單 worker
這次只動它同時在線 users 數：10 → 50 → 100，每級各跑一輪
  1. 全套起來：docker compose up -d --build --wait
    為什麼：demo1 就是 compose 的預設腳本，三容器（db / 靶機 / Locust）一起上、cpuset 與記憶體限制全套生效，不必另外指定 LOCUSTFILE。
  2. 開 Web UI http://localhost:8089/?tab=charts，先用 10 users 開打
    為什麼從 10 開始：低負載先確認管線是通的、報表會動，當作後面兩級的基準；一上來就 100 users，「慢」是真瓶頸還是還沒暖機會分不清。
  3. 依序 50、100 users 各跑一輪，每輪記錄 p50 / p95 / RPS / error rate
    為什麼一次只加一級：users 是這支唯一動的旋鈕。三級分開跑，才能畫出「users → RPS」的關係，看出它不是線性。
# 全套起來（db + 靶機 + Locust），demo1 是預設腳本
docker compose up -d --build --wait
open http://localhost:8089/?tab=charts   # UI 設 10 → 50 → 100 users 各跑一輪，直接看 Charts
結果 / 關鍵洞察

100 users 配 wait_time(1,3)，RPS 不會衝到 100，而是卡在 ≈ 50：每個 user 做完一個請求要先「思考」平均 2 秒，100 ÷ 2 ≈ 50 RPS。RPS 是吞吐量、users 是併發數，兩者不相等——這是讀懂後面所有報表的第一課。

02 locustfiles/demo2_event_loop.py

Event Loop 卡死實況 — 實驗 A / B / C

 一個 time.sleep 凍結全服務，healthz 也死▶

上面模擬器的真槍實彈版。用同樣 20 users、同樣 constant(0)（連發）打三個只差在「怎麼等那 1 秒」的端點，把 blocking / async / CPU-bound 一次量出來。靶機只有 1 worker、1 條 event loop，這是所有戲劇的前提。

實驗設計 — 三組對照，變因是「端點怎麼等」
固定不動20 users、wait_time constant(0)、靶機單 event loop
這次只動它打哪個端點 = 等待方式：async sleep / blocking sleep / CPU 計算
◆ 對照組 B — 正常基準 /async-sleep

await asyncio.sleep：等待時 event loop 切去服務別人。RPS ≈ 20、p95 ≈ 1s。

◆ 實驗組 A — blocking /sync-sleep

time.sleep：卡死唯一的 event loop，全服務凍結。RPS ≈ 1、p95 數十秒。

◆ 實驗組 C — CPU bound /cpu?n=30

純計算：宣告 async def 也無法切換。RPS ≈ 10、全部排隊。

  1. 先跑對照組 B：在 class-picker 勾 AsyncSleepUser，20 users / spawn 20，按 Start
    為什麼先跑 B：它是「正確行為」的基準線——RPS≈20、p95≈1s。手上先有「正常長什麼樣」，A 和 C 的異常才有對照。
  2. Stop 後改勾實驗組 A：SyncSleepUser（users / spawn 不變）
    為什麼只換 class：users、時長、spawn rate 全固定，唯一變因是端點等待方式，p95 從 1s 暴衝到數十秒就只能歸因到 blocking。
  3. 跑 A 的同時，另開 terminal：curl -m 5 http://localhost:8000/healthz
    為什麼這一刀最關鍵：/healthz 根本不碰 DB、理應毫秒回應，此刻卻 timeout——證明 blocking 卡的不是「某個請求」，是整條服務。這正是 K8s liveness probe 會誤殺 Pod 的根因。
  4. 最後改勾實驗組 C：CpuUser
    為什麼留到最後：打破「async = 快」的迷思——C 是 async def 卻照樣串行，因為 CPU 計算期間沒有讓出點。async 解決 I/O 等待，救不了 CPU bound。
# 起 Web UI，在 class-picker 選 class 開打（教學預設用 UI 看 Charts）
docker compose up -d --build --wait
LOCUSTFILE=demo2_event_loop.py docker compose up -d locust
open http://localhost:8089/?tab=charts   # 先勾 AsyncSleepUser(B)，20 users / spawn 20，Start
# 看完一輪按 Stop → 改勾 SyncSleepUser(A)、CpuUser(C) 各跑一輪對照
結果 / 三組對照（20 users）
組別端點RPSp95
對照 Basync-sleep≈ 20≈ 1s
實驗 Async-sleep≈ 1數十秒
實驗 Ccpu≈ 10全排隊

同樣 20 users，只因「怎麼等」不同，吞吐差 20 倍。blocking 與 CPU-bound 都會獨佔單一 event loop；async 只在 I/O 等待時有效。

03 locustfiles/demo3_attacker_bottleneck.py

攻擊機自己先陣亡 — HttpUser vs FastHttpUser

 5,921 → 24,762 RPS：換武器差 4.2 倍▶

做任何靶機實驗前的前置檢查：先證明攻擊機自己不是瓶頸，否則 demo4/5/6 全是在測 Locust，不是測靶機。同樣 300 users、同樣打 /healthz（靶機最快、幾乎零成本的端點），只換攻擊機用的 HTTP client。

實驗設計 — 變因是「攻擊機的武器」，靶機完全不動
固定不動300 users、端點 /healthz、constant(0)、靶機設定一律不碰
這次只動它攻擊機 HTTP client（武器）：requests ↔ geventhttpclient ↔ 多 process
◆ 對照組 — 舊武器 SlowAttacker（HttpUser）

基於 requests，純 Python 開銷。單一 locust process 的 CPU 會先鎖死 100%（GIL 限制）。

◆ 實驗組 — 新武器 FastAttacker（FastHttpUser）

基於 geventhttpclient（C 實作）。同樣 300 users 吞吐翻數倍；再加 --processes 4 吃滿 4 核又翻一次。

  1. 另開 terminal 先掛上 docker stats
    為什麼要先開：本 demo 重點不在 RPS 數字，而在攻擊機的 CPU%。沒同時盯攻擊方，看到「RPS 上不去」會誤判成靶機弱。
  2. 第一輪對照組：class-picker 勾 SlowAttacker，300 users / spawn 100，Start
    觀察什麼：locust 容器 CPU 鎖死 ~100%（單核），RPS 卡在 5,921。此刻去看靶機 uvicorn——CPU 還很閒。瓶頸在攻擊機，數據是假的。
  3. Stop 後改勾實驗組：FastAttacker，users / spawn 不變
    為什麼參數要完全一樣：唯一變因是武器。RPS 從 5,921 跳到 24,762（4.2×），靶機沒任何改動——直接證明剛才壓的是 Locust 的天花板。
  4. 第三輪 --processes 4：要停掉預設 locust、改用 --service-ports 起（見指令；前菜：階段三分散式）
    為什麼還要加 process：單 process 受 GIL 限制只吃一核；locust 容器有 4 核（cpuset 4-7），開 4 process 才吃得滿。RPS 再翻——接著瓶頸換人（靶機？網路？）。
docker compose up -d --build --wait
LOCUSTFILE=demo3_attacker_bottleneck.py docker compose up -d locust
open http://localhost:8089/?tab=charts   # 勾 SlowAttacker 跑一輪 → Stop → 改勾 FastAttacker，皆 300 users / spawn 100
docker stats   # 全程盯 locust 容器 CPU% —— SlowAttacker 會先鎖死 100%

# 第三輪要 --processes 4：停掉預設 locust 讓出 8089，改用一次性 run（不加 --headless = Web UI）
docker compose stop locust
docker compose run --rm --service-ports locust -f /mnt/locust/demo3_attacker_bottleneck.py \
  --host http://app:8000 --processes 4 --class-picker
open http://localhost:8089/?tab=charts   # 勾 FastAttacker，Start
結果 / 已驗證（300 users・/healthz）
武器RPS瓶頸落點
SlowAttacker5,921攻擊機 CPU 100%（單核）
FastAttacker24,762攻擊機，但已大幅鬆綁

同參數差 4.2×——差的全是攻擊機自己。判斷瓶頸在攻擊機的訊號：靶機 CPU 還很閒，RPS 卻上不去。

04 locustfiles/demo4_sync_vs_async_db.py

sync vs async 打真 PG — 核心對照組

 同 100 併發：async 512 RPS vs sync 335 RPS▶

本訓練的核心對照實驗。前置都做完了（會用工具、懂 event loop、攻擊機不是瓶頸），現在打真 PostgreSQL。?delay_ms=100 讓每個查詢佔住 connection 100ms，模擬慢查詢。

實驗設計 — 變因只有併發模型，其餘全部釘死
固定不動同 driver psycopg3、同 pool=100、100 users、delay_ms=100
這次只動它create_engine（sync）↔ create_async_engine（async）
◆ 對照組 — 傳統 sync /sync/posts

FastAPI 把它丟進 threadpool（預設 40 條）跑，每條 thread 佔住一個連線等 DB。瓶頸 = 40 條 thread。

◆ 實驗組 — async /async/posts

等 DB 時 event loop 切去服務別人，不佔 thread。瓶頸往後推到 pool（100）或 CPU。

  1. export DB_POOL_SIZE=100，再 docker compose up -d --build --wait
    為什麼先把 pool 開大：要讓 sync 的瓶頸確定落在 threadpool（40）而不是 pool。pool 太小兩組都卡 pool、看不出差異（demo5 會證明）。為什麼一定要 export：不 export，下一個 compose 指令會用預設 pool=5 把 app 默默重建回去——真踩過，四輪數據一模一樣。
  2. 先跑對照組：class-picker 勾 SyncPostsUser，100 users / spawn 50，Start
    為什麼當對照：sync 是「沒做任何特別處理」的基準。先量它，才知道 async 到底快多少。
  3. Stop 後改勾實驗組：AsyncPostsUser，users / spawn 不變
    為什麼只換 class：唯一變因是併發模型。100 併發 > threadpool 40 時，sync 被迫排隊、async 不必，差異此刻才現形。
  4. 跑完驗證旋鈕：docker inspect k8s-app-1 … | grep DB_POOL
    為什麼養成這習慣：確認 app 真的吃到 pool=100 而非被默默重建回 5。「改了變因卻沒變化，先驗證旋鈕接上了」是 LESSON 04。
# pool 開大，逼 sync 瓶頸落在 threadpool（40）而非 pool
# 一定要 export，否則下個指令以預設 pool=5 把 app 重建回去（真實踩過）
export DB_POOL_SIZE=100
docker compose up -d --build --wait
# 起 Web UI，對照組 → 實驗組只在 class-picker 換 class，users / spawn 不變
LOCUSTFILE=demo4_sync_vs_async_db.py docker compose up -d locust
open http://localhost:8089/?tab=charts   # 勾 SyncPostsUser → Stop → 改勾 AsyncPostsUser，皆 100 users / spawn 50
# 驗證旋鈕真的生效
docker inspect k8s-app-1 --format '{{.Config.Env}}' | tr ' ' '\n' | grep DB_POOL
結果 / 已驗證（pool=100・100 users）
組別端點RPS瓶頸
對照 sync/sync/posts335threadpool 40 條
實驗 async/async/posts512pool / event loop CPU

pool 夠大時 async 領先（512 vs 335 RPS）。但領先幅度被 ORM 序列化的 CPU 成本吃掉一截（理論 3× → 實測 ~1.5×，LESSON 03）——把 -u 拉到 200/400，看 sync 的 p95 線性惡化、async 撐到 pool 滿才惡化。

05 locustfiles/demo5_pool_starvation.py

Pool Starvation — DB_POOL_SIZE 10 → 100 掃描

 167 → 642 RPS：pool 翻倍 p95 砍半，直到 CPU 102%▶

固定攻擊參數，只掃一個旋鈕 DB_POOL_SIZE，畫出「p95 對 pool 大小」的曲線。這支證明：connection pool 才是 p95 的主宰，async 不是魔法。?delay_ms=50 讓查詢佔住連線、放大 pool 的影響。

實驗設計 — 四輪掃描，變因只有 pool 大小
固定不動200 users、delay_ms=50、打 /async/posts、每輪重置 DB
這次只動它DB_POOL_SIZE：10 → 20 → 50 → 100，四輪
◆ 對照組 — 基準 DB_POOL_SIZE = 10

pool 嚴重不足，請求在 pool 前大排長龍。此時 sync ≈ async（1.28s ≈ 1.28s）——瓶頸在 pool，誰都救不了。

◆ 實驗組 — 逐級放大 20 / 50 / 100

每翻一倍看 p95 怎麼走。一路砍半，直到靶機 CPU 撞 102%、瓶頸從 pool 轉移到 event loop 為止。

  1. 每輪開頭先重置：docker compose down
    為什麼每輪都重置（鐵則 03）：不掛 volume，down 即回到同一份 seed（50 users / 200 posts）。每輪從相同起點出發，四條 p95 數據才能放同一張圖比較。
  2. 設這一輪 pool：export DB_POOL_SIZE=10，再 up -d --wait
    為什麼從 10 當對照：先量「pool 嚴重不足」長什麼樣，當基準線。為什麼 export：同 demo4——不 export 會被下個 compose run 默默重設回 5，四輪全是假數據（LESSON 04 的由來）。
  3. 打固定攻擊：UI 設 200 users / spawn 100，跑 ~60s 記錄 RPS / p95
    為什麼攻擊參數一字不動：四輪唯一變因是 pool。攻擊一變，p95 的變化就無法單純歸因到 pool。
  4. 換 export DB_POOL_SIZE=20 / 50 / 100，重複步驟 1–3
    為什麼掃四級而非只比兩點：要看出「報酬遞減」的轉折——pool 50→100 幾乎不再有效益，代表瓶頸已換人。一條曲線比兩個點更有說服力。
# 每輪：重置 DB（鐵則 3）→ 換 pool 重啟 → 同一個攻擊 → 記錄
docker compose down
export DB_POOL_SIZE=10    # 下一輪換 20 / 50 / 100，攻擊參數不動
docker compose up -d --build --wait
LOCUSTFILE=demo5_pool_starvation.py docker compose up -d locust
open http://localhost:8089/?tab=charts   # UI 設 200 users / spawn 100，跑 ~60s 看 Charts 的 p95

兩支探針，兩個問題

livenessProbe → /healthz：「死了嗎？」失敗就重啟 container。 刻意不碰 DB——如果 liveness 打 DB，pool 耗盡時 K8s 會誤判 App 死了而瘋狂重啟， 那不是 App 的問題，是探針設計錯誤。

readinessProbe → /readyz：「準備好了嗎？」真打一次 DB， 失敗只是暫時從 Service 摘掉、不重啟。DB 還沒好，Pod 就不收流量。

demo2 的 K8s 進階版：sync-sleep 卡死 event loop → 連 /healthz 都回不了 → liveness 連續 fail → 重啟 → 新 Pod 起來又被打 → 又重啟——這就是 K8s 下的雪崩迴圈。 模擬器（03）演的那一幕，在 K8s 裡會被 kubelet 放大。

RPS 低 ~14%、p50 反而略好——差異來自攻擊機在 host 與 VM 共擠 + kube-proxy/LoadBalancer 多一跳 + CFS 無核心隔離。靶機 CPU 648m，離 limit 2000m 還遠。 量級一致 → 單 Pod baseline 成立，「1000 撐住、5000 必崩」就是階段三 HPA 的對照組。

踩坑 01 · 真實案發K8s 的 command: 蓋掉的是 ENTRYPOINT，args: 才對應 Docker 的 CMD。把 compose 的 command 原樣搬過來 → docker-entrypoint.sh 被跳過 → init.sql 沒跑、PG 以 root 啟動被拒 → db 和 app 一起 CrashLoopBackOff。
踩坑 02 · 真實案發RollingUpdate 讓 DB 連線足跡翻倍。新舊 Pod 並存的瞬間，pool 預熱撞上 too many clients → 新 Pod CrashLoop、舊 Pod 不退 → rollout 永久死鎖。本 demo 用 Recreate 治標，production 正解是 pgbouncer（→ LESSON 05）。
踩坑 03改 ConfigMap 不會自動生效。configMapKeyRef 的 env 只在容器啟動時讀一次，必須 kubectl rollout restart 才收斂——驗證方式：kubectl exec ... env | grep DB_POOL_SIZE（LESSON 04 的 K8s 版：先驗證旋鈕接上了）。
踩坑 04kubectl patch 是暫時的。patch 是 imperative 操作，下次 kubectl apply 會把 ConfigMap 蓋回 YAML 宣告的值——declarative 才是真相，實驗完想保留設定就改 manifest 再 apply。
部署實戰 — 八步上線到清理，每步都附「為什麼現在做」
  1. ① 前置 · build image：docker compose build app，確認 kubectl config current-context 是 orbstack
    為什麼現在做：OrbStack 的 K8s 跟 Docker 共用同一個 image store，build 完那顆 image kubelet 直接看得到、免推 registry——這正是 app.yaml 用 imagePullPolicy: Never 的前提。沒先 build，Pod 會卡在 ErrImageNeverPull。
  2. ② 部署 · 按依賴順序 apply：先 namespace.yaml，再 kubectl apply -f k8s/
    為什麼有順序：namespace 是其他資源的容器，必須先存在；其餘同目錄一起 apply，K8s 會自理建立順序。db.yaml 一檔含四物件（ConfigMap 裝 init.sql、Secret 裝帳密、Deployment、ClusterIP Service）。
  3. ③ 驗證 Pod · kubectl get pods -n loadtest -w 等到 1/1 Running
    為什麼盯 Ready 而不只看 Running：db 的 readiness 用 pg_isready、app 的 readiness 打 /readyz 真連一次 DB——app 要等 DB 通了才會 Ready，這是刻意的啟動順序保護。卡住就 kubectl describe pod / logs 看事件。
  4. ④ 驗證 Service · 拿 LoadBalancer 的 EXTERNAL-IP，curl …/healthz
    為什麼分兩種 Service：db 是 ClusterIP（只在叢集內、不對外＝正確）；app 是 LoadBalancer（OrbStack 自動配 EXTERNAL-IP 給 host 直打）。IP 還沒分配就先 kubectl port-forward svc/app 8000:8000 頂著。
  5. ⑤ 開打 · host 上 uv run locust … --host http://$APP_IP:8000 --class-picker，開 :8089/?tab=charts
    為什麼 Locust 跑 host、不進 K8s：沿用攻防分離；也因此 K8s baseline 比 compose 多了「攻擊機與 VM 共擠 + LoadBalancer 一跳」的成本（對答案見上方 baseline）。換 demo 只換 -f，靶機不必重部署。
  6. ⑥ 監控 · kubectl top pod -n loadtest（docker stats 的 K8s 版）
    為什麼一定要盯（鐵則 01）：單位是 millicpu（648m = 0.648 vCPU）。重點看 app 和 db 會不會擠到同一顆核心——CFS quota 只限「CPU 時間」、不鎖核，這是和 compose cpuset 最大的行為差異。
  7. ⑦ 實驗旋鈕 · 改 pool 要三步：patch ConfigMap → rollout restart → exec 驗證
    為什麼不能只 patch：configMapKeyRef 的 env 只在容器啟動時讀一次，patch 完 Deployment 模板沒變 → 不觸發 rollout → 跑著的 Pod 還拿舊值（踩坑 03）。必須 rollout restart 收斂，再 exec … | grep DB_POOL_SIZE 驗證旋鈕真的接上（LESSON 04 的 K8s 版）。注意 patch 是暫時的，下次 apply 會蓋回（踩坑 04）。
  8. ⑧ 清理 · kubectl delete namespace loadtest 一刀全清
    為什麼這樣等於重置：namespace 是隔離邊界，刪它連帶刪光裡面所有物件。db 用 emptyDir（揮發），下次 apply 時 init.sql 重跑、回到 50 users / 200 posts 的 seed——這就是鐵則 03「每輪重置 DB」的 K8s 版。
